# Scenarios

最后核对：2026-07-22

每个场景记录：任务特征、首选模式、候选工具和边界、上下文/数据/权限、验证方式、成本/失败/安全风险，以及数小时或数天可完成的最小试验。

先按任务结构选模式，而不是先选产品：步骤和分支稳定时优先确定性 workflow；一个人能顺序完成时用单 Agent + Skills；只有子任务能独立验收时才增加 subagent；需要大范围、可并行探索且漏项代价很高时才考虑 swarm；跨小时或跨天时再加入持久化状态、checkpoint 和恢复。模型提出的越权、超出已验证包络、不可逆或高影响动作需要具体人工批准；已风险评估且处于预授权确定性流程内的低风险动作可以按策略自动执行。人工批准是治理闸门，不是功能安全措施。

## 调研与知识工作

### 任务特征

- 典型任务：技术选型、竞品/市场扫描、论文综述、故障资料汇总、标准与官方文档核对。
- 难点不只是“找到很多链接”，而是限定时间截面、区分一手/二手来源、处理互相冲突的证据，并让每条结论可回溯。
- 只有在问题可按互不依赖的来源域或子问题切开时才有真实并行，例如“官方文档、论文、独立评测、安全规范”四条检索线。

### 首选模式，以及为什么不先采用更复杂模式

首选 **单 Agent + 调研 Skill + 动态检索工具 + 证据台账**。先让一个上下文持有研究问题、纳入标准和结论结构，减少重复检索和口径漂移；来源足够多且分区明确时，再派只负责“找证据、不下总判断”的 subagent，主 Agent 负责合并和冲突裁决。

不先用 swarm：同模型、同搜索入口生成的“多人意见”容易共享盲点；并行数增加后，去重、来源核验和综合成本也会上升。若一个 Agent 在数小时内能覆盖问题，swarm 通常只是更贵的检索器。

### 候选工具和边界

- Codex、Claude Code 或支持联网检索/文件读写的同类 Harness：适合维护 Markdown 证据表、拉取原文并生成带引用的草稿；最终判断仍需人工看原始来源。
- 浏览器搜索、官方站点、arXiv/OpenReview、GitHub release/API：是数据来源，不等于验证器；摘要和搜索片段不能替代原文。
- MCP 搜索、数据库或知识库连接器：适合统一接入内部资料；MCP 只解决连接，不负责来源质量、任务拆分或多 Agent 协作。
- LangGraph 等状态图框架：只在调研跨天、需要增量抓取、人工复核队列和断点恢复时采用；一次性报告不值得先引入运行时。

### 上下文、数据与权限

- 明确研究问题、截止日期、地区/版本、术语定义、纳入/排除标准和期望输出。
- 为每条证据保存标题、URL、发布者、发布日期、检索日期、原文位置、证据等级和限制；结论与证据分表，避免把摘录直接当结论。
- 默认只读访问互联网和内部知识库；内部文档按数据分级过滤，禁止把密钥、客户数据或未公开资料发送给外部模型。
- 外部网页视为不可信输入，网页中的“忽略前述指令”等内容不得改变研究任务或授权范围。

### 验证方式

- 对关键结论逐条打开原始链接，确认主体、日期、适用版本和原文是否真的支持该结论。
- 至少抽查所有高影响结论和具体数字；数字同时记录比较基线、样本、Harness/实验设置和厂商是否自评。
- 独立寻找反证或失败案例；若只有厂商材料，标为“厂商自报/待外部验证”，不提升为普遍规律。
- 运行链接检查、重复来源检查和字段完整性检查；最终摘要中的每个事实都能反向定位到证据台账。

### 成本、失败与安全风险

- 成本主要来自重复检索、长文上下文和多 Agent 汇总；先限制问题和每条检索线的预算。
- 常见失败是引用不存在、把发布日期当事件日期、把预印本当定论、引用链循环，以及多个 Agent 互相抄同一二手来源。
- 外部内容可能包含 prompt injection、恶意附件或追踪链接；调研 Agent 不应拥有 shell 写权限、邮件发送权或生产系统凭证。
- 版权资料只保留必要短摘录和定位，不批量复制原文。

### 最小试验（约 4 小时）

- **交付物**：选一个具体问题，例如“ROS 2 Jazzy 是否适合作为未来两年的作品集基线”，用单 Agent 建立 10 条一手/独立来源的证据表，再派一个 subagent 只找反证，最后输出一页决策备忘录。
- **验收条件**：每条重要结论都有可访问 URL、核对日期、证据等级和限制；至少一条反证；人工抽查 5 条无错引；总 token/时间有记录。
- **面试价值**：证明的是可审计技术调研和工程决策能力，不是“会让多个 Agent 搜索”。

## 编码与大型仓库

### 任务特征

- 工作对象包含代码、测试、依赖、生成文件、CI、数据库迁移和多个仓库之间的契约；真正难点是定位影响面、保持改动隔离并提供可执行验证。
- 并行上限由依赖图和文件所有权决定，不由“最多能启动多少 Agent”决定。

| 任务形态 | 首选模式 | 进入并行/长任务的触发条件 | 必须保留的边界 |
| --- | --- | --- | --- |
| 单仓库、单功能或缺陷 | 单 Agent + 仓库 Skill + 测试驱动/系统化调试 | 一个 Agent 无法同时持有多个独立技术域，且各域有独立验收 | 先读 `AGENTS.md`、测试和改动边界；不得让多个 Agent 同改一组文件 |
| 独立模块并行 | 协调 Agent + 少量 subagents + 独立 worktree/目录所有权 | 模块契约已冻结、无共享写热点、每个模块可单独测试 | 主 Agent 先定义接口和验收；集成者统一处理冲突与回归 |
| 跨仓库迁移 | 显式任务图 + 每仓库负责人 + 人工集成闸门 | 提供方、消费者、部署或数据迁移能按兼容阶段推进 | 先做兼容矩阵和发布顺序；禁止各 Agent 私自改变跨仓契约 |
| 后台长任务 | 持久化后台 Agent/workflow + checkpoint + 预算/超时 | 全量测试、索引、迁移或大规模机械改写跨小时/跨天 | 保存提交 SHA、任务状态和中间产物；恢复前确认工作树和依赖未漂移 |

### 首选模式，以及为什么不先采用更复杂模式

默认从 **单 Agent + 仓库级 Skills + 本地工具链** 开始。它最容易保持统一上下文、遵守现有架构和从失败测试追到修复。只有任务图出现两个以上“输入已知、写集不重叠、验收独立”的就绪节点时才并行；跨仓库和长任务则升级为显式 workflow，而不是自由聊天的 Agent 群。

不先用 swarm：共享文件、隐含接口和统一测试会把名义并行变成冲突与返工；同一模型复制多个角色也不会自动产生独立审查证据。

### 候选工具和边界

- Codex、Claude Code、OpenHands 等 Coding Harness：适合仓库检索、编辑、命令执行和迭代测试；能力比较必须连同 Harness、工具权限与提示规则一起看。
- `rg`、LSP、AST/类型检查器、测试框架、lint、构建系统和 Git：这些确定性工具是首选事实来源；代码图谱/MCP 可缩小导航范围，但不能替代编译与测试。
- Git worktree、CI job matrix：适合写集隔离和模块级验证；worktree 隔离不等于接口兼容。
- LangGraph、Temporal 或队列型自研 runtime：适合需要持久化、重试、审批和恢复的跨仓迁移；小改动采用它们会增加状态模型与运维成本。

### 上下文、数据与权限

- 提供仓库指令、目标行为、非目标、相关 ADR/接口、允许修改的路径、测试命令和验收标准。
- 并行前为每个 Agent 指定输入提交 SHA、文件/模块所有权、输出格式、依赖和禁止修改项；共享事实写入契约文件或任务台账，不依赖聊天记忆。
- 默认在沙箱和最小文件权限中运行；读取秘密、联网安装依赖、推送分支、改 CI/云资源需单独授权。
- 跨仓迁移保存 API/schema 版本、消费者清单、向前/向后兼容要求、发布与回滚顺序。

### 验证方式

- 修改前先得到失败测试、复现步骤或可观察基线；修改后运行最窄测试，再运行受影响模块的类型检查、lint、构建和回归测试。
- 验证器尽量使用独立证据：编译器、测试、静态分析、浏览器截图、性能基准、契约测试和人工 code review，而不是让生成代码的 Agent 再说“看起来正确”。
- 合并并行分支后必须在统一工作树重跑集成测试；跨仓库按兼容矩阵验证旧/新提供方与消费者组合。
- 长任务每个 checkpoint 记录提交 SHA、已完成节点、命令、退出码、产物哈希、剩余预算和恢复条件。

### 成本、失败与安全风险

- 成本包括上下文扫描、重复构建、并行 Agent token、合并冲突和 CI 资源；每增加一个 Agent 都要证明缩短了关键路径。
- 常见失败：多个 Agent 改同一文件、测试桩互相矛盾、只跑新增测试、生成依赖锁文件噪声、任务恢复到错误 SHA。
- 供应链风险包括恶意依赖、仓库内 prompt injection、测试脚本执行任意命令和密钥泄漏；安装、发布、推送和生产部署必须受审批保护。
- 对数据库/协议迁移，代码回滚不代表数据可回滚；必须单独设计 expand/migrate/contract 或等价兼容路径。

### 最小试验（1 至 2 天）

- **交付物**：准备两个难度和验收条件相近、都包含前端、API 和测试的小功能，从各自相同的干净提交开始并统一预算。一个由单 Agent 完成，另一个把类型/后端、前端适配、测试补充画成候选节点，只并行真正无共享写集的节点；第二轮交换任务形态以减小任务与学习顺序偏差，并记录任务图、耗时、token、冲突和测试结果。
- **验收条件**：每个节点有独立测试；最终统一构建通过；没有未授权路径改动；在统一预算和质量指标下比较两轮结果，不因一次实验就宣称普遍并行收益。
- **面试价值**：证明大型仓库任务拆解、契约管理和 CI 验证能力，比展示“同时开了几个 Agent”更可信。

## 产品设计与前端实现

### 任务特征

- 典型任务从需求、信息架构、交互状态、设计系统映射一路到响应式页面、可访问性和真实浏览器验证。
- 输入常常不完整且视觉结果具有主观性；“代码能编译”不能证明布局、文案、状态和交互正确。
- 设计与实现高度耦合，尤其是复用既有组件库时，过早分派多个 Agent 容易产生互相不兼容的视觉语言。

### 首选模式，以及为什么不先采用更复杂模式

首选 **单产品 Agent + 设计/前端 Skills + 浏览器或 Figma 工具 + 人工视觉检查**，让同一上下文持有用户目标、设计系统、交互状态和实现约束。只有桌面/移动端验证、独立页面或可访问性审计可以独立验收时，才把这些工作交给 subagent。

不先用 swarm：视觉方向需要一致决策，多个“设计角色”常制造方案数量而非用户证据；主流程尚未确定时并行实现会放大返工。

### 候选工具和边界

- Figma 与 Code Connect：适合读取/维护设计系统和组件映射；自动生成节点不能替代变量、状态、响应式和可访问性设计。
- Coding Harness + React/Vue/Angular 本地工具链：适合在既有工程中实现；必须服从仓库组件库和样式规范。
- Playwright、浏览器 DevTools、截图和像素检查：用于真实渲染、交互和控制台验证；视觉截图不能替代语义化 DOM 与键盘测试。
- 图像生成工具：只在确实需要位图素材时使用；不应用生成图掩盖真实产品信息或替代既有品牌资产。

### 上下文、数据与权限

- 必须准备目标用户、核心任务、已有页面/组件、设计 token、支持的视口、浏览器、状态清单、文案来源和真实/脱敏数据。
- 先列 loading、empty、error、disabled、permission denied、long text 和 slow network 等状态，再实现主路径。
- Figma 写权限、图片版权、埋点数据和后端写操作分别授权；默认使用本地 mock，不连接生产写接口。
- 对内部产品截图和客户数据做脱敏；从设计稿提取的访问令牌不得进入仓库。

### 验证方式

- 在至少一个桌面和一个移动视口运行 Playwright，检查首屏、溢出、遮挡、滚动、菜单/对话框、表单校验和控制台错误。
- 做键盘导航、焦点、语义标签、颜色对比和 reduced-motion 基础检查；用组件/端到端测试验证关键状态。
- 将实现与设计系统变量、组件变体和交互说明逐项对照；对没有设计依据的变化明确标注为工程决策。
- 邀请目标用户或产品负责人完成一个核心任务；行为证据优先于 Agent 自评和静态截图。

### 成本、失败与安全风险

- 成本主要在反复截图、浏览器运行、素材生成和设计返工；先冻结核心工作流，再补视觉细节。
- 常见失败是只实现 happy path、用假数据掩盖溢出、桌面正常但移动端重叠、组件另起一套样式、视觉回归无人核对。
- 浏览器自动化可能误点真实提交/删除按钮；端到端测试使用隔离账号和测试环境，对有副作用动作增加拦截或人工批准。
- 生成文案与图像存在事实、版权和品牌风险；上线前由内容/品牌负责人审核。

### 最小试验（1 天）

- **交付物**：做一个“机器人设备状态与告警侧栏”，覆盖桌面/移动端以及 loading、empty、offline、alarm、permission denied 五种状态，使用现有组件库并写 Playwright 截图/交互测试。
- **验收条件**：两个视口无重叠与横向溢出；键盘可完成筛选和查看详情；控制台无错误；状态测试与人工走查通过。
- **面试价值**：把前端优势转成机器人 HMI/数字孪生可检查的产品交付证据。

## 运维与业务自动化

### 任务特征

- 输入是工单、告警、邮件、表格或业务事件，输出可能触发通知、改配置、退款、建单、扩容或远程操作。
- 规则通常大部分稳定、少部分需要非结构化理解；失败会造成重复执行、越权、资金损失或不可追踪变更。
- 任务跨系统、跨分钟或跨天，天然需要状态、超时、重试、补偿和人工队列。

### 首选模式，以及为什么不先采用更复杂模式

首选 **确定性 workflow 编排 + 单 Agent 做分类/抽取/解释 + human-in-the-loop 审批高风险动作**。让代码决定分支、重试、预算、幂等和权限，让模型处理语言不确定性并输出结构化候选结果。

不先用自治 swarm：业务步骤已知时，多 Agent 对话没有增加控制保证，反而扩大凭证面和重复副作用风险。只有告警需要多个独立专业域诊断且各自只读时，才并行收集候选解释。

### 候选工具和边界

- Temporal、LangGraph 等持久化工作流：适合跨时任务、checkpoint、重试和人工恢复；必须理解它们各自的重放/副作用语义。
- n8n、Zapier 等低代码自动化：适合低风险 SaaS 连接和快速试验；复杂状态、强一致或安全关键流程不宜只靠画布配置。
- OpenAI Agents SDK、Google ADK 等 Agent runtime：适合受控 tool calling、handoff 和 tracing；它们不自动提供业务幂等、授权和回滚。
- MCP/原生 API 连接器：只暴露必要的只读查询或窄写操作；连接成功不代表有权调用。

### 上下文、数据与权限

- 将业务政策、字段 schema、风险等级、允许动作、审批人、SLA、最大重试和升级路径写成机器可检查规则。
- 使用短期、按动作分 scope 的凭证；读写工具分开，生产和测试连接器分开，高风险工具不进入普通 Agent 的可见工具集。
- 每次运行保存输入版本、模型/提示版本、工具参数、返回摘要、审批人和最终结果；个人信息按最小必要原则进入模型。
- 对外部邮件、工单正文和附件执行不可信输入隔离，不能让其改变系统提示或授权策略。

### 验证方式

- 用历史脱敏样本建立离线集，分别测分类正确率、字段抽取、策略违规、工具参数和拒绝率；不要只测“最终文本看起来合理”。
- 在 shadow/read-only 模式运行，对比人工决策；高影响误报和漏报分别统计。
- 对每个写动作验证幂等 key、超时、有限重试、重复消息、并发冲突和补偿路径；故障注入覆盖 API 5xx、超时和部分成功。
- 审计记录能从业务事件追到模型判断、审批和实际 API 结果。

### 成本、失败与安全风险

- 成本来自每事件推理、跨系统调用、人工审批队列和 trace 存储；先做规则过滤，只把不确定样本交给模型。
- 常见失败是无限重试、重复退款/通知、模型把缺字段补成事实、审批疲劳、外部文本注入和凭证过宽。
- 回滚不是一句提示词：对不可逆动作使用预授权额度、双人批准、补偿事务或根本不开放自动执行。
- 模型/提示升级可能改变行为，必须版本化并在历史样本上回归。

### 最小试验（1--2 天）

- **交付物**：对 50 条脱敏运维告警做“规则过滤 -> Agent 结构化归类/建议 -> 人工批准 -> 只写测试工单”的 workflow，加入稳定幂等 key、超时、重试上限和审计记录。
- **验收条件**：重复投递不重复建单；字段 schema 全部校验；任何生产动作不可达；能从一次运行恢复；人工可看到证据并拒绝建议。
- **面试价值**：证明的是 Agent 与可靠工作流、权限和可观测性的整合，而不是聊天机器人包装。

## 工业软件与设备集成

### 任务特征

- 连接 PLC、相机、机器人、传感器或 MES/SCADA，面对设备状态、告警、配方、命令和网络抖动；物理世界不会因模型“解释合理”而回滚。
- 协议数据有采样周期、质量位、时间戳、单位和设备生命周期，命令有前置条件、互锁、确认和完成反馈。
- 重点是状态机、幂等、超时、回滚/补偿、审计和降级，不是让 LLM 替代设备控制程序。

### 首选模式，以及为什么不先采用更复杂模式

首选 **确定性设备适配层 + 显式状态机/workflow + Agent 只读诊断和候选操作说明 + 人工批准**。协议驱动、状态转换、互锁、重试和命令执行由普通代码/PLC 完成；LLM 可以把日志、手册和历史工单整理成诊断假设。

不先用多 Agent 自治执行：多个 Agent 无法提供实时性、互锁或功能安全保证，还会扩大凭证与竞态面。仅当机械、电气、视觉、网络日志可以独立分析时，才并行产生只读诊断结果，再由统一状态模型合并。

### 候选工具和边界

- OPC UA、MQTT、Modbus 模拟器与各自成熟客户端：用于协议和状态数据；LLM 不直接拼接原始报文或决定写寄存器。
- XState/SCXML、Temporal 或项目既有状态机/工作流引擎：管理命令生命周期、超时、重试和补偿；实时闭环和安全互锁仍留在 PLC/控制器。
- Grafana、OpenTelemetry、时序库和事件日志：提供独立可观测证据；Agent 可查询摘要，但不能改写原始审计记录。
- RAG + MCP/窄 API：适合只读访问设备手册、告警码和历史维修单；MCP server 必须做认证、参数校验、速率限制和工具白名单。

### 上下文、数据与权限

- 建立 canonical device model：设备 ID、连接态、运行态、模式、告警、质量位、单位、时间戳和命令关联 ID；不能只把原始 tag 值塞进 prompt。
- 每个命令定义允许源状态、目标状态、参数范围、互锁、幂等 key、超时、确认/完成事件、取消、补偿和人工权限。
- 分离只读遥测、低风险配置和运动/能量相关命令；Agent 默认只有仿真或只读权限，PLC/安全控制器凭证永不暴露给模型。
- 审计至少记录操作者/Agent、设备、命令、参数摘要、策略版本、审批、时间、结果和相关 trace；敏感工艺与客户数据脱敏。

### 验证方式

- 从协议模拟器开始，运行状态转换测试、契约测试和 trace replay；覆盖断线重连、乱序/重复消息、过期数据、质量位异常和部分成功。
- 对所有写命令验证幂等：同一 key 重放不产生第二次物理意图；非幂等动作默认不自动重试。
- 用故障注入验证超时、取消、补偿和人工接管；回滚不可行时，设计进入已知安全状态并升级，而非假装恢复原状。
- Agent 诊断用带标签历史案例离线评测，证据引用必须指向原始告警/手册；执行层测试与 Agent 自评独立。

### 成本、失败与安全风险

- 成本主要在建立统一状态模型、模拟器、协议测试和审计设施；这是系统可靠性的必要投入，不应被 prompt 工程替代。
- 常见失败是把连接态当运行态、忽略数据质量/时序、对非幂等命令重试、多个控制源争用、回滚路径从未演练。
- prompt injection 可能来自设备名、工单、日志或手册；这些全部是不可信数据。Agent 输出必须过 schema、策略和状态机三重校验。
- LLM 延迟和可用性不得进入实时控制关键路径；模型不可用时系统应保留监控、确定性控制和人工操作能力。

### 最小试验（2 天）

- **交付物**：用 MQTT 或 OPC UA 模拟三态设备（`OFFLINE/IDLE/RUNNING`）和告警，做只读诊断 Agent；唯一“执行”工具只写仿真命令队列，状态机验证前置条件、稳定幂等 key、超时和审计。
- **验收条件**：重复命令不重复生效；非法状态转换 fail closed；断线/乱序/超时可重放；每条建议含数据时间和证据；不存在真实设备凭证。
- **面试价值**：证明设备抽象、可靠状态流和 AI 辅助诊断，是工业应用软件岗位的直接证据；不声称具备产线功能安全认证。

## 机器人应用与 AI 辅助运维

### 任务特征

- 任务同时包含自然语言意图、机器人状态、TF/坐标系、规划、运动执行、反馈、取消、告警和安全互锁。
- LLM 擅长解释非结构化意图、检索手册、汇总状态和提出候选计划，但不提供硬实时、运动学正确性、碰撞避免或功能安全保证。
- 一个看似简单的“移动到工位”也必须经过目标消歧、坐标/单位校验、规划、约束检查、批准、执行反馈和异常处理。

### 首选模式，以及为什么不先采用更复杂模式

首选 **LLM 意图/检索/候选计划/解释层 + 确定性任务编排与验证层 + ROS 2/控制器执行层 + 人工批准**：

1. LLM 只把自然语言转换为受限 intent schema，检索 SOP/手册，给出带证据的候选步骤并解释状态。
2. 行为树或显式状态机管理任务顺序、重试、恢复和降级；规划器、运动学/碰撞检查器验证候选目标和轨迹。
3. ROS 2 action 负责可取消、带反馈的长动作；驱动、机器人控制器、PLC 和独立安全控制器负责执行、互锁和急停。
4. 人工在仿真结果、目标、速度/区域和风险摘要可见后批准高风险动作。

不先用 swarm：更多 LLM 不能替代 TF、规划器、控制器或安全 PLC 的确定性证据，也不能通过“多数同意”证明轨迹安全。多 Agent 仅适合并行读取机械、视觉、网络和任务日志，输出只读诊断假设。

### 候选工具和边界

- ROS 2 Jazzy topics/services/actions、TF2、URDF、rosbag：提供机器人软件事实和可重放数据；action 接口仍需服务端验证 goal、支持 cancel 并报告 feedback/result。
- BehaviorTree.CPP/Nav2 BT 或项目状态机：负责任务编排与恢复；它们不是功能安全控制器，也不替代底层规划和控制。
- MoveIt 2/Nav2/厂商规划器与仿真器（Gazebo、Webots、MuJoCo、Isaac Sim 选一）：负责约束、碰撞、轨迹或环境仿真；LLM 不能凭文本绕过规划失败。
- RAG、MCP 或窄只读 API：查询手册、参数、告警、TF 摘要和最近 trace；初始试验不得暴露 `/cmd_vel`、关节命令、PLC 写入或安全控制器接口。

### 上下文、数据与权限

- intent schema 只允许白名单任务、命名实体和单位；目标由场景 ID/语义位置解析成受版本管理的 frame/pose，不接受模型任意生成控制报文。
- 提供 URDF/SRDF、TF 快照、关节限制、工具/TCP、工作区、安全区域、规划参数、机器人模式和数据时间戳；缺项或数据过期时 fail closed。
- 机器人写权限与知识库读取权限分离；仿真、测试台和生产命名空间/凭证彻底隔离。急停、安全门和速度监控不经过 LLM。
- trace 记录原始意图、解析结果、检索证据、候选计划、规划结果、审批、action goal/feedback/result、取消和异常。

### 验证方式

- 第一阶段只读：用 rosbag/仿真状态验证 Agent 是否正确解释当前模式、告警和 TF，不开放任何命令工具。
- 第二阶段仅仿真：schema 校验后由确定性代码生成 goal，规划器检查可达性、关节限制和碰撞；测试取消、超时、stale state、frame 不存在和规划失败。
- 将自然语言解析、计划正确性、规划可行性、执行结果和安全拒绝分开计量；绝不能用“最终到达”掩盖中间越权或不安全尝试。
- 红队测试 prompt injection、单位混淆、相似设备名、过期状态、重复请求和绕过批准；验证任一不确定条件都会拒绝执行。

### 成本、失败与安全风险

- 成本集中在数字孪生、状态/坐标规范、仿真、规划和 trace，而不是模型调用；这些也是对机器人岗位最有价值的工程证据。
- 常见失败是 frame/单位错误、状态过期、模型虚构对象、行为树无限重试、取消未传播、仿真与实机参数漂移，以及把普通软件检查误称为功能安全。
- LLM 不在实时控制闭环中，不负责急停、碰撞保护、安全限速和硬件互锁；任何生产运动都需风险评估、现场验证和符合组织/法规的安全系统。
- 模型不可用、输出非法或证据不足时，系统必须保持停止/只读/人工操作等已定义降级状态。

### 最小试验（2--3 天，仅仿真或只读）

- **交付物**：构建“ROS 2 机器人状态解释器”：读取固定 rosbag 或仿真 topics、TF 和 action feedback，RAG 检索两份手册，输出结构化告警解释与候选恢复步骤。可选的“执行”按钮只发送到 mock action server 或仿真，并需人工批准。
- **验收条件**：无真实机器人/PLC 写接口；每条解释引用时间戳与数据源；未知 frame、过期状态、非法单位、规划失败和取消均 fail closed；trace 可完整重放一次任务。
- **面试价值**：证明 ROS 2 数据流、TF、action、可观测性与安全分层；不宣称具备实机运动控制、全身控制或安全认证能力。
