# Flow Graph 教程重构设计

## 目标

将现有 Flow Graph 长篇教程重构为两篇适合作者阅读和实践的 Hexo 笔记。教程使用 TypeScript 和通用工业视觉流程，先完成可运行的最小执行器，再解释可视化编辑、流式执行和工业安全。

最终笔记不得出现内部项目名称、项目版本代号或架构决策编号。通用技术架构、设计理由、源码结构和工程决议可以保留。

## 读者与定位

目标读者具备多年 Web 全栈和前端开发经验，熟悉 TypeScript、异步编程和前端数据流，但正在补充机器人软件、工业视觉和设备系统知识。

教程不把读者当作图论或机器人专业的新生，也不假定其已经掌握图执行器。解释优先利用以下已有认知：

- TypeScript 类型、函数参数和返回值。
- 前端组件的 props 和事件。
- Promise 依赖、状态管理和表单校验。
- React 可视化界面与后端运行语义的边界。

## 内容方案

采用案例驱动的渐进方案。两篇文章使用同一个虚构工业视觉流程：

```text
相机采集 -> 缺陷检测 -> 计算调整量 -> 安全确认 -> 模拟执行
```

案例只用于讲清数据契约、调度和安全边界，不对应任何真实公司、客户、设备或生产参数。

### 上篇：最小执行器

暂定标题：《Flow Graph 入门：用 TypeScript 实现工业视觉流程执行器》。文章提供完整、可复制的最小实验代码，内容顺序如下：

1. 用函数参数和组件 props 类比 typed port。
2. 区分 DAG、typed dataflow 和状态机。
3. 定义节点、端口、边、图文档和运行上下文。
4. 建立节点注册表，使编辑器、校验器和执行器共享契约。
5. 实现加载期校验：节点、端口、类型、必填输入和环。
6. 编译拓扑顺序和边绑定，生成静态执行计划。
7. 实现确定性执行器和最小 trace。
8. 运行工业视觉示例并观察输出。
9. 用测试锁定成功与失败语义。

上篇暂不实现分支、并行、流式队列、重试和真实设备调用。结尾明确这些限制，并指向下篇。

### 下篇：可视化、流式与安全

暂定标题：《Flow Graph 进阶：可视化、流式执行与工业安全》。文章保留关键接口、伪代码、Mermaid 图和工程检查表，不重复一套完整工程代码，内容如下：

1. React Flow handles 如何映射 typed ports。
2. 编辑模型、编译模型和运行模型为什么必须分离。
3. fan-out、fan-in、barrier、branch 和 skipped 语义。
4. 批式 DAG 与流式 Flow 的区别。
5. 有界队列、背压、顺序和关联 ID。
6. 取消、超时、有限重试和 trace。
7. L1 原子节点与 L2 复合节点。
8. Mock、simulate/real 模式和硬件能力注入。
9. 单一安全闸门、工位 barrier 和异常时拒绝执行。
10. 可检查的项目验收清单和面试价值。

## 教学表达

每个核心概念采用固定的四层表达：

1. **直觉**：先用读者熟悉的 Web/TypeScript 概念建立类比。
2. **数据流**：用小型 Mermaid 图展示输入、处理和输出。
3. **代码**：说明代码解决了什么，以及尚未解决什么。
4. **验收**：提供运行命令、预期输出、失败示例或练习。

每张图只引入一个主要概念。术语第一次出现时同时给出中文解释，后文才使用英文缩写。代码示例避免为了展示泛型技巧而增加不必要的类型复杂度。

## 架构决议的表达

架构决议保留通用结论，但不引用内部决策编号。必须提及时采用统一结构：

```text
背景问题 -> 考虑过的方案 -> 最终选择 -> 选择理由 -> 代价
```

教程需要准确表达以下决议及其演进：

### 从隐式黑板到 typed edges

共享上下文可保存 run ID、取消信号和日志能力，但业务数据不应依赖字符串后缀、全表扫描或隐式变量名。节点产物应通过显式 typed ports 和 edges 传递，错误尽量提前到加载期。

### 编辑模型与执行模型分离

编辑器保存完整的端口图、布局和配置。运行前将图校验并编译为拓扑顺序和直接边绑定。运行时不依赖画布坐标，也不需要模拟一个高开销的分布式消息队列。大数据通过 handle 或引用传递。

### 原子节点与复合节点分层

L1 原子节点用于组合和测试，L2 复合节点封装高频流程与受控能力。不是所有内部步骤都应该作为可获得真实硬件权限的自由组合节点。

### 教学节点不等于真实硬件能力

`register`、`adjust`、`execute` 可用于模拟模式理解数据流，但真实纠偏不能允许用户通过任意连线绕过安全判断。真实能力必须收敛到受控复合节点和单一 runner。

### 数值闸门与工位 barrier 正交

工位 barrier 判断数据是否齐备以及设备是否到位；数值 gate 判断调整量是否可接受。两者解决不同风险，真实动作需要同时满足。

### 计算与派发分离

批量流程先计算并保留可观察的中间产物，再经过 barrier 统一派发。计算节点默认无设备写入能力，派发节点通过受控接口执行真实副作用。

## 错误与安全语义

上篇区分 Parse、Validation、Compile 和 NodeExecution 四类错误。校验错误必须能定位节点、边或端口；执行错误必须包含 run ID 和节点 ID。

下篇补充以下规则：

- `simulate` 是默认模式，不触发真实硬件动作。
- `real` 模式必须显式注入 runner 和所需设备能力。
- 缺 runner、输入不完整、工位未到位或安全 gate 拒绝时 fail-closed。
- 取消信号必须传到真实 I/O，不能只在外层放弃等待。
- 自动重试只用于明确可重试且幂等的操作。
- trace 记录摘要、状态和耗时，不记录密钥或不必要的原始敏感数据。
