# Agent 工作模式地图

最后核对：2026-07-22

这里的“模式”描述任务怎样被拆分、执行、校验和接管，不是产品排行榜。同一个项目通常会组合多种模式，例如：确定性状态机负责设备流程，单 Agent 负责解释告警，Skills 固化排障步骤，高风险动作必须人工批准。

## 先按任务特征选择

| 任务特征 | 首选模式 | 升级信号 | 不要先做什么 |
| --- | --- | --- | --- |
| 目标清楚，主要难点是跨文件或跨工具推理 | Single Agent with Dynamic Tools | 上下文开始互相污染，或出现三个以上独立调查面 | 不要仅为“更智能”先上多 Agent |
| 团队反复做同一种工程流程 | Skills-driven Agent | Skill 内出现持久状态、审批或大量分支 | 不要把凭证、运行时和外部服务塞进 Skill |
| 有少量边界清楚、可独立验收的子任务 | Subagents | 子任务数量动态增长，且需要分层委派 | 不要让多个 Agent 同时改同一文件 |
| 搜索空间宽、分片同构、并行收益可测 | Agent Swarm | 任务需要稳定 SLA、精确重试和事务语义 | 不要把 swarm 用于串行依赖链或设备控制 |
| 步骤、分支、重试和审计要求明确 | Deterministic Workflow | 某一步需要开放式判断或非结构化材料处理 | 不要让 LLM 自由改写业务状态机 |
| 单次运行跨分钟、小时或更久 | Background and Long-running Agents | 到达发布、付费、写设备或安全边界 | 不要依赖一个不断增长的聊天上下文 |
| 错误代价高、目标含主观判断或权限升级 | Human-in-the-loop | 同类决策稳定且已有可量化误差边界 | 不要把“用户点一下确认”当完整安全设计 |

建议的升级顺序是：先证明单 Agent 可完成，再把重复流程提炼成 Skill；只有确有上下文隔离或并行收益时才引入 subagents；只有任务宽度、预算和聚合规则都可测时才考虑 swarm。对工业软件，确定性 workflow 和 Human-in-the-loop 往往是外层，Agent 只是其中受限的一步。

## 分层模型：不要把名词混成能力

```text
Model
  -> Agent loop
    -> Skills
      -> Harness
        -> MCP / native tools
          -> A2A
            -> Multi-agent workflow
```

这是一条从推理核心到系统协作的观察路径，不是强制的调用栈；一套实现可以跳过某层，也可以在同层选择多种组件。

| 层级 | 回答的问题 | 典型产物 | 不能据此推出 |
| --- | --- | --- | --- |
| Model | 下一段内容或工具调用如何生成？ | 模型权重、API、推理参数 | 模型分数不能单独证明任务能端到端完成 |
| Agent loop | 何时思考、调用工具、观察结果、停止？ | ReAct 循环、状态与停止条件 | 有循环不等于有可靠重试、权限和恢复 |
| Skills | 某类任务的可复用做法如何按需注入？ | `SKILL.md`、脚本、模板、参考资料 | Skill 是流程知识，不是工具协议或执行沙箱 |
| Harness | 模型、上下文、工具、权限和验证怎样装配？ | Codex、Claude Code、KimiCode、定制运行器 | 同一模型换 harness 可能得到不同结果，不能混报 |
| MCP / native tools | Agent 怎样访问数据和动作？ | MCP server、函数调用、CLI、浏览器、文件系统 | MCP 解决连接，不决定怎样拆任务或多 Agent 协作 |
| A2A | 独立 Agent 服务怎样发现、委派、回传和异步协作？ | Agent Card、task、artifact、消息 | A2A 不是内部规划算法，也不替代工具访问控制 |
| Multi-agent workflow | 多个角色怎样分工、同步、仲裁和收敛？ | supervisor、graph、swarm、事件流 | Agent 数量多不自动带来正确性或速度 |

最容易犯的三类错误：把“支持 MCP”写成“支持多 Agent”；把一组 prompt/Skill 写成可恢复的 workflow；把模型在特定 harness 下的成绩归因给裸模型。选型和评测时至少同时记录模型、harness、工具集、推理预算、并发、数据集版本和验证规则。

## Single Agent with Dynamic Tools

- **替代的旧做法：** 替代人工在搜索、终端、编辑器、数据库和工单系统之间复制信息，也替代为每个任务预先写死一条调用链。
- **触发条件：** 目标能够由一个负责人保持整体一致性，但解决路径要根据观察结果动态选择；例如定位 ROS2 topic 延迟、修改 Web 调试台并运行测试。
- **上下文组织：** 稳定规则放仓库说明；当前目标、约束和验收条件放任务上下文；大文件和日志通过搜索后局部读取；工具返回结构化摘要，不把全量输出持续堆入窗口。
- **并行方式：** 默认串行观察再决策；只并行无依赖的只读读取、搜索和测试，写操作保持单写者。
- **验证方式：** 要求 Agent 运行可复现命令，核对 diff、测试、类型检查或接口响应；最终结论附文件位置或命令证据。
- **主要成本：** 长上下文和工具往返产生 token、延迟与 API 成本；动态路径使复现和预算上限比脚本困难。
- **不适用情形：** 大量同构独立分片、跨小时运行、严格事务流程，或一次错误就可能伤人/损坏设备的控制回路。
- **最小试验：** 选一个 30 分钟内可人工完成的仓库 bug，限定只读诊断工具和一个可写目录，比较 Agent 的成功率、工具调用数、耗时以及测试是否真正覆盖修复。

## Skills-driven Agent

- **替代的旧做法：** 替代散落在 wiki、聊天记录和个人经验中的 SOP，以及每次都重复粘贴的超长系统提示。
- **触发条件：** 同一类任务已经重复出现，步骤顺序、检查项或领域约束趋于稳定，但具体输入和工具选择仍需判断。可把“重复三次”作为开始提炼的经验起点，而不是通用实证阈值；高风险流程应更早固化，快速变化的流程则应更晚。
- **上下文组织：** Skill 入口只保留触发描述；主体写流程与验收；大段参考资料、脚本和模板按需加载，形成渐进式上下文，而不是启动时全量注入。
- **并行方式：** Skill 本身不创造并行；它可以规定哪些只读调查可并行、怎样切分 subagent，以及写操作如何串行合并。
- **验证方式：** 用典型、边界和反例任务检查触发准确率、步骤遵守率和产物验收；Skill 版本与任务结果一起记录。
- **主要成本：** 编写、维护、去重和跨 harness 适配；过度具体会快速过时，过度抽象则退化成无效提示。
- **不适用情形：** 一次性探索、需求仍剧烈变化、必须持久调度/事务恢复，或核心问题其实是缺少一个可调用工具。
- **最小试验：** 把“ROS2 故障采集”写成一个 Skill，只包含 topic/TF/日志检查顺序、输出模板和禁止动作；在两次不同故障中比较漏项数和上下文用量。

## Subagents

- **替代的旧做法：** 替代主 Agent 在一个上下文中轮流扮演研究员、实现者和审查者，也替代人工开多个聊天窗口再手工汇总。
- **触发条件：** 存在少量边界清楚、依赖少、可独立交付摘要或 patch 的任务，且隔离上下文能减少干扰；例如前端、ROS bridge 和测试分别调查。可先以 2 到 8 个子任务作为经验起点，再按委派开销、验收能力和平台并发限制调整，而不是追求固定 Agent 数量。
- **上下文组织：** 主 Agent 只下发目标、文件范围、接口和验收；子 Agent 自己读取所需材料，返回结构化结论、证据和未决问题；共享事实落到文件或 artifact，不靠转述记忆。
- **并行方式：** 调研和不同文件所有权可并行；共享接口先约定，重叠文件使用单写者或串行交接；主 Agent 负责依赖排序与汇总。
- **验证方式：** 每个子任务有局部验收，汇总后再跑跨模块测试；独立 reviewer 检查规格符合性和实现质量，但不能以“另一个 Agent 同意”代替测试。
- **主要成本：** 重复读取仓库、上下文扇出、协调和合并冲突；小任务的委派成本可能高于执行成本。
- **不适用情形：** 任务强串行、接口尚未定义、所有修改集中在同一小文件，或没有主 Agent 能判断子结果是否正确。
- **最小试验：** 将一个小功能拆为“API 契约核对、UI 实现、测试设计”三个只读/分文件任务，记录相对单 Agent 的总 token、墙钟时间、冲突数和集成测试结果。

## Agent Swarm

- **替代的旧做法：** 替代人工批量分派大量搜索、文档扫描、候选生成和多视角审查，也替代固定数量、固定角色的手工多 Agent 编排。
- **触发条件：** 任务具有高宽度：分片数量大或未知、分片相对独立、可以动态扩缩容，并且有明确的去重、聚合和停止标准。
- **上下文组织：** 根任务保留目标与预算；协调者生成分片和角色；工作 Agent 只拿最小材料；结果进入共享 artifact/index，聚合者读取摘要与证据，不把所有原始轨迹塞回根上下文。
- **并行方式：** 多层动态派生、批量并发和分阶段 map-reduce；设置最大深度、并发宽度、总工具调用和超时，避免无界扩张。
- **验证方式：** 先测覆盖率、重复率、引用有效率和分片失败率，再抽样核对原始证据；关键结论要求交叉验证，聚合产物用确定性规则检查完整性。
- **主要成本：** token 和工具调用迅速放大，外部 API 限流、重复劳动、错误传播和汇总瓶颈都更明显；可观测性与预算控制是前置条件。
- **不适用情形：** 串行依赖、单一事实查询、共享状态频繁写入、低延迟交互、精确事务，以及机器人实时控制或安全联锁。
- **最小试验：** 对 30 份设备手册按文件并行提取故障码，固定最多 5 个工作 Agent 和 100 次工具调用；与单 Agent 比较字段召回、错误引用、总成本和墙钟时间。

Kimi 在 [K2.5 Agent Swarm 官方博客](https://www.kimi.com/blog/agent-swarm)披露“最多 100 个 subagents、超过 1,500 次工具调用、相对串行执行快 4.5 倍”。这是厂商对其产品场景的披露，不是独立、跨系统、可直接外推的通用基准；使用它只能证明 swarm 的工程尺度示例，不能证明 100 个 Agent 是最优规模。

## Deterministic Workflow

- **替代的旧做法：** 替代靠人工记住分支、重试和补偿的操作流程，也替代让 LLM 每次自由决定业务关键步骤。
- **触发条件：** 流程图、状态转移、超时、重试、幂等和审计可提前定义；只有少数节点需要模型处理非结构化输入。
- **上下文组织：** 业务状态保存在数据库或 workflow runtime；每个 Agent 节点只接收类型化输入并返回类型化结果；大 artifact 使用引用，不把聊天记录当状态库。
- **并行方式：** 由 DAG、状态机或事件规则明确 fork/join；并行分支有幂等键、超时、重试和补偿，不由模型临时猜测依赖。
- **验证方式：** 单元测试状态转移，集成测试外部适配器，故障注入测试重试/恢复；模型节点另做离线数据集评测并设置信心阈值。
- **主要成本：** 前期建模和维护状态机，处理 schema 演进、补偿逻辑与运行时基础设施；灵活性低于开放式 Agent。
- **不适用情形：** 目标和路径都未知的探索性研究，或一次性的开放创作任务。
- **最小试验：** 实现“告警进入 -> 拉取上下文 -> Agent 归因 -> 人工批准 -> 创建工单”五步流程，模拟超时和重复消息，验证不会重复建单且能从中断点恢复。

## Background and Long-running Agents

- **替代的旧做法：** 替代人持续盯住终端、定时刷新任务和手工续跑，也替代一个前台 HTTP 请求承载数小时工作。
- **触发条件：** 任务跨越分钟到天、需要等待外部事件、可分检查点，并允许异步查看进度、取消、暂停和恢复。
- **上下文组织：** 计划、检查点、artifact、权限租约和事件日志外置持久化；每次唤醒重建最小上下文；摘要可更新但原始证据保持可追溯。
- **并行方式：** 队列分发独立 job，按资源和 API 限流控制并发；使用心跳、租约和幂等键处理 worker 丢失及重复投递。
- **验证方式：** 检查中间 artifact、心跳与预算；最终执行 acceptance suite；通过故障注入验证重启、取消、超时和重复投递。
- **主要成本：** 持久化、调度、沙箱、密钥轮换、日志与人工运营；运行越久，外部环境变化和权限漂移越难控制。
- **不适用情形：** 需要毫秒级反馈、任务不可检查点化、无限开放目标，或无人值守期间会执行不可逆高风险动作。
- **最小试验：** 让后台 Agent 每小时读取一次模拟设备日志并更新报告，运行 8 小时；中途杀死 worker，验证恢复后无重复告警且预算、取消和审计均可见。

## Human-in-the-loop

- **替代的旧做法：** 替代“全自动或全人工”的二选一，也替代在事故后才人工复核的被动治理。
- **触发条件：** 涉及发布、付费、删除、外部通信、设备写入、安全边界、模糊目标或低置信度判断；人与 Agent 各自有明显比较优势。
- **上下文组织：** 审批界面展示拟执行动作、原因、关键证据、diff、风险、权限范围和可回滚方案；保留 Agent 轨迹引用，但不要求审批者阅读全部思维过程。
- **并行方式：** Agent 可并行准备候选和证据，所有高风险分支在审批闸门汇合；同一动作只生成一个具幂等键的执行请求。
- **验证方式：** 自动规则先挡明显错误；人工只审批具体动作而不是抽象意图；执行后再核对实际结果，并记录拒绝、修改和覆盖原因用于评测。
- **主要成本：** 等待时间、审批疲劳、责任边界和 UI/审计开发；闸门过多会使人机械点击，反而失效。
- **不适用情形：** 高频实时控制、审批者没有足够证据或专业能力，以及已经能由确定性规则安全自动化的低风险步骤。
- **最小试验：** 在 AI 辅助机器人运维 demo 中只允许 Agent 生成诊断和 ROS2 命令草案；操作者核对目标设备、命令、预计影响和回滚后才能执行，统计拒绝率、修改率与误放行。

## 组合原则：面向工业与机器人软件

```text
设备/ROS2 数据 -> 确定性采集与状态机
              -> 受限 Agent + 排障 Skill
              -> 可并行只读调查（必要时 subagents）
              -> 证据、命令草案、影响范围
              -> 人工批准
              -> 确定性执行器 -> 反馈与审计
```

对作者当前方向，最有面试价值的不是展示最多 Agent，而是展示五件事：状态外置、工具最小权限、证据可追溯、失败可恢复、高风险动作可拦截。只有当单 Agent 基线已经记录且任务确实可分片，subagents 或 swarm 的速度数字才有意义。
