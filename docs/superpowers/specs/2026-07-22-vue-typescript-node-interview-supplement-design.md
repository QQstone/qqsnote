# Vue、TypeScript 与 Node.js 面试知识补充设计

## 目标

在保留原始学习笔记的前提下，补充资深软件开发工程师面试中常见的 Vue、TypeScript 与 Node.js 基础知识，并新建一篇统一复习索引。补充内容应帮助作者解释底层机制、比较方案、分析故障和说明工程取舍，而不是继续堆积 API 清单或背诵式问答。

## 读者与定位

- 主要读者是具有 11 年 Web/full-stack 经验、前端能力较强的作者本人。
- 内容定位是资深工程师基础复习材料，不是初学者教程，也不是框架源码逐行解析。
- 回答深度以能够应对原理追问、工程场景追问和方案权衡为准。
- 示例优先连接实时看板、Web3D、工业设备状态和机器人调试界面等作者可能面对的业务，但不把这些示例扩展成新的项目教程。

## 当前内容判断

- Vue 文章主要记录 Vue 2 和早期 Vue 3 的语法、生命周期与组件通信，缺少 Vue 3 响应式机制、渲染调度、组件边界和性能诊断。
- `VueRouter.md` 只有 front matter，尚不能支撑路由原理、权限和缓存相关面试。
- `TypeScript.md` 已覆盖基础类型、接口、类、泛型和部分高级类型，但内容偏语法清单，部分表述需要用现代 TypeScript 的类型系统视角补充边界。
- Node.js 内容分散在 Express、进程、流、包管理和运行时文章中。现有 `nodejs-runtime.md` 过短，`node-stream.md` 只有类型简介，缺少事件循环、背压、故障隔离和资源治理等资深岗位常问内容。
- 旧文章仍具有学习轨迹价值，因此本次不整体重写，不静默替换其表达。

## 内容组织方案

采用“原文就地补充 + 独立总索引”的组合方式：

1. 在五篇最相关的现有文章末尾增加主题补充，读者可以在原知识上下文中深入复习。
2. 新建一篇总索引，提供知识地图、推荐复习顺序、自测题和跨栈工程场景。
3. 总索引只总结和导航，不复制各主题文章中的完整解释与代码。

这种组织方式保留了旧笔记，同时避免把所有内容塞进单篇文章，也避免为同一知识点维护两份正文。

新建总索引本身是完整的新文章，不使用补充边界标记。

## 单节写作模板

每个重要知识点按以下顺序组织，按复杂度删减，不机械填满所有小节：

1. **核心结论**：先给可直接回答面试官的一到三句话。
2. **机制解释**：说明运行过程、关键约束或类型推导。
3. **最小示例**：给可运行或可独立推理的代码，避免伪代码依赖未声明变量。
4. **工程取舍**：解释何时使用、何时不用，以及可靠性或性能代价。
5. **常见追问**：列出一至三个问题，并给回答要点而非背诵长文。
6. **回答边界**：指出常见误解，例如路由守卫不是服务端授权、TypeScript 类型不会自动校验运行时数据。

## 技术口径

- Vue 内容以 Vue 3、Composition API 和 Vue Router 4 为主；Vue 2 内容只在说明迁移差异时出现。
- 不把 Vue 的响应式依赖追踪简化成“数据变化就重新渲染整个页面”。
- 不把 `key` 简化成单纯的性能优化，它首先用于表达节点身份。
- TypeScript 内容以 `strict` 模式为默认讨论前提，明确类型系统存在不健全性和类型擦除。
- 不使用 `any` 掩盖示例中的类型问题；外部输入先视为 `unknown` 再校验。
- Node.js 事件循环顺序只描述稳定、可验证的语义，不依赖容易随版本变化的过细实现细节。
- 不把 Node.js 说成“所有工作都在一个线程”；区分 JavaScript 执行线程、libuv 线程池和操作系统异步 I/O。
- 性能建议先测量事件循环延迟、CPU、内存和 I/O，再选择缓存、流、Worker 或多进程方案。
